# Project: unefy — Modern Club & Association Management

Open-source, self-hostable Vereinsverwaltung with optional SaaS deployment. Built with a Next.js frontend, FastAPI backend, and native mobile apps (iOS/Android).

## Architecture Principles

### Multi-Tenancy & Deployment
- **Tenant isolation** is a first-class concern — every data model must be tenant-scoped
- The same codebase runs in both **self-hosted** (single-tenant) and **SaaS** (multi-tenant) mode
- Deployment mode is determined by environment config, not code branches
- Self-hosted deployments must work with a single `docker compose up`
- Never hard-code SaaS-specific logic — use feature flags / config for SaaS-only features (billing, onboarding, etc.)

### Multi-Client Architecture
- **Backend API is the single source of truth** — consumed by Web (via Next.js BFF), native mobile apps, and public API clients
- The API must be fully functional standalone — never assume a Next.js BFF is the only consumer
- Three API audiences:
  1. **Internal API** (`/api/v1/...`): Used by Next.js BFF and mobile apps (JWT auth, full CRUD)
  2. **Public API** (`/api/public/v1/...`): For third-party integrations and user-built tools (API key auth, rate-limited, documented with OpenAPI)
  3. **Webhook endpoints**: For incoming events from payment providers, etc.

### Monorepo Structure
```
unefy/
├── apps/
│   ├── web/          # Next.js frontend (BFF for web)
│   └── mobile/       # Native apps: Swift/SwiftUI (iOS) + Kotlin/Compose (Android)
├── backend/          # FastAPI backend (serves all clients)
├── docker/           # Docker & compose files
├── docs/             # Architecture docs, specs, public API docs
└── CLAUDE.md         # This file
```

### Communication
- Web frontend communicates with backend via Next.js Server Actions (BFF pattern)
- Mobile apps communicate directly with the backend REST API
- Public API consumers use API keys with scoped permissions
- All API responses follow a consistent envelope: `{ data?: T, error?: { code: string, message: string } }`

### Authentication Strategy

**Auth is split by client type — each uses the best mechanism for its platform.**

| Client | Auth Method | Session/Token Storage |
|--------|------------|----------------------|
| **Web** | better-auth (Next.js) — manages session cookies | Opaque session cookie (httpOnly, Secure, SameSite=Lax) → session state in DB/Redis |
| **Mobile** | OAuth 2.0 + PKCE → JWT access + refresh tokens | iOS Keychain / Android Keystore |
| **Public API** | API key (`X-API-Key` header) | Consumer-managed |

**MVP Auth Methods:**
1. **Magic Link / Email OTP** — passwordless, primary login method
2. **Google OAuth** — social login
3. **Passkeys / WebAuthn** — modern passwordless, phishing-resistant

**Roadmap (post-MVP):**
- Apple OAuth (for iOS app launch)
- Email + Password (traditional fallback)
- MFA / TOTP
- OIDC / external IdP support (for self-hosted enterprise: Keycloak, Authentik, Azure AD)
- Biometric unlock (mobile)
- Device management & trusted devices
- Impersonation (admin feature for SaaS support)

**Key principle:** Web auth is handled by `better-auth` in the Next.js layer. The backend trusts the session and provides a verified user context. Mobile and Public API authenticate directly against the backend's JWT/API-key endpoints.

## Git Conventions

### Commits
- Small, focused commits — one logical change per commit
- Conventional commits with scope: `type(scope): description`
- Types: `feat`, `fix`, `refactor`, `chore`, `docs`, `style`, `test`
- Scopes: `members`, `events`, `dues`, `auth`, `billing`, `ui`, `api`, `db`, `config`, `docs`, `public-api`
- English, lowercase, imperative mood. First line max 72 chars.
- Examples:
  - `feat(members): add member profile page with edit form`
  - `feat(events): implement event registration flow`
  - `feat(dues): add automatic payment reminder emails`
  - `fix(auth): handle expired refresh token gracefully`
  - `feat(public-api): add member list endpoint with filtering`
  - `refactor(api): extract pagination into reusable dependency`
  - `chore(config): add docker-compose for local development`
- Do NOT mention AI assistants, Claude, LLM, or any AI tool in commit messages, code, or comments
- Do NOT add "Generated by" or "Co-authored-by" trailers
- Commit as the repository owner (use existing git config)

### Branching
- Work directly on `main` for now (solo project phase)
- Feature branches + squash merge for larger features

## Quality Assurance & Review Process

### Testing Philosophy
- **Tests are mandatory** — no feature is complete without tests
- Test behavior, not implementation details
- Every bug fix requires a regression test
- Tests must be fast enough to run on every commit

### Testing Pyramid

| Level | What | Where | Runner |
|-------|------|-------|--------|
| **Unit** | Services, utilities, validation logic | Backend: `pytest`, Frontend: `vitest` | Fast, no I/O |
| **Integration** | API endpoints, Server Actions, DB queries | Backend: `pytest` + real PostgreSQL, Frontend: `vitest` | Real DB, mocked externals |
| **E2E** | Critical user flows (login, member CRUD, event registration) | Playwright | Real browser, full stack |

### Coverage Requirements

| Area | Minimum Coverage |
|------|-----------------|
| Auth logic (backend + frontend) | 100% |
| Services / business logic | 80% |
| API endpoints | 80% |
| Server Actions | 80% |
| UI components | Critical interactions only |
| E2E flows | All happy paths + key error paths |

### Code Review Process

Every change goes through a structured review before merging — even in solo development.

#### 1. Pre-Commit Checks (automated, via pre-commit hooks)
```
Backend:  ruff check + ruff format + mypy
Frontend: eslint + prettier + tsc --noEmit
```
These MUST pass before a commit is created. No `--no-verify`.

#### 2. Pre-Push Checks (automated, via pre-push hook or CI)
```
Backend:  pytest (full suite)
Frontend: vitest run + npm run build
```

#### 3. PR-Based Review (for all non-trivial changes)
- Work on feature branches, merge via PR
- PR must include:
  - Clear description of what and why
  - Test coverage for new/changed behavior
  - Screenshots for UI changes
- Automated CI must pass before merge (see CI Pipeline)
- Self-review checklist before requesting merge:
  - [ ] Tests pass locally
  - [ ] No `console.log`, `print()`, debug artifacts
  - [ ] No hardcoded secrets or credentials
  - [ ] Tenant isolation maintained (backend)
  - [ ] Loading states / skeletons for async components (frontend)
  - [ ] Error handling for all failure cases
  - [ ] i18n for all new UI text
  - [ ] Migration reviewed if DB changes (backend)

#### 4. AI-Assisted Review
When generating code with Claude or other AI tools:
- **Always review generated code** — never blindly commit
- Run the full test suite after AI-generated changes
- Verify tenant scoping on any new DB queries
- Check for security issues (injection, auth bypass, data leakage)
- Ensure generated code follows the patterns in these CLAUDE.md files

### CI Pipeline (GitHub Actions)

Runs on every push and PR:

```yaml
# Backend
- ruff check + ruff format --check
- mypy --strict
- pytest --cov --cov-fail-under=80
- alembic check (verify no pending migrations)

# Frontend
- eslint
- tsc --noEmit
- vitest run --coverage
- next build (catches build errors)

# E2E (on PRs to main only)
- Playwright tests against docker-compose stack

# Security
- dependency audit (pip-audit / npm audit)
```

**CI must be green before any merge to main. No exceptions.**

### Branch Protection (when ready)
- `main` branch protected: no direct pushes
- Require CI to pass
- Require at least 1 approval (when team grows) or self-review confirmation (solo)
- Squash merge to keep history clean

## Code Conventions

### Language
- All code in English: file names, variables, functions, types, comments
- UI-facing text in German (primary audience: DACH region clubs/associations)
- All UI strings must use i18n from day one — no hardcoded text in components
- Keep translation files organized by feature/module
- API error messages in English (machine-readable codes + human-readable messages)

### General
- Prefer simple, readable code over clever abstractions
- No premature optimization — make it correct first
- Every new API endpoint needs input validation
- Write types/interfaces for all data structures shared between frontend and backend
- Type hints everywhere (Python + TypeScript strict mode)
